## 一、现状概览

### 1.1 agent-swarm 当前架构

```
┌──────────────────────────────────────┐
│           run_examples/              │  使用层
├──────────────────────────────────────┤
│   MainRollout  /  SubRollout         │  编排层
├──────────────────────────────────────┤
│        Agent + AgentConfig           │  智能体层
├──────────────────────────────────────┤
│   tool/        │    swarm_tool/      │  工具层
│  (Search,      │  (CreateSubagent,   │
│   CodeRunner)  │   TaskTool)         │
├──────────────────────────────────────┤
│   utils/ (LLMClient × 3 种后端)      │  通信层
└──────────────────────────────────────┘
```

**核心特点**：
- 编排器-工作器模式（MainRollout 协调，SubRollout 执行）
- 异构模型支持（主/子智能体可用不同模型，如 kimi-k2.5 主 + qwen 子）
- 防递归设计（子代理工具列表排除 swarm_tool）
- 步数管理（SubRollout 默认 20 步，带步数提示和剩余步数警告）
- 三种 LLM 客户端后端（标准 OpenAI / trpc 网关 / eval API）

### 1.2 Anthropic 生产系统架构

```
用户查询 → LeadResearcher → 制定策略 → 并行创建 Subagents
                ↑                              ↓
                └─ 综合/迭代 ← 子代理并行搜索 ─┘
                         ↓
                  CitationAgent → 带引用的最终结果
```

**核心特点**：
- 编排器-工作器模式 + 专用引用代理
- 外部 Memory 存储（防止上下文截断）
- 并行工具调用（缩短研究时间达 90%）
- 动态缩放（按查询复杂度调整子代理数量）
- 代理自我改进（使用 LLM 优化自身提示）

## 二、多智能体架构的必要性论证

### 2.1 为什么 agent-swarm 需要多智能体？——四个核心论据

#### 论据一：单代理的上下文窗口瓶颈

Anthropic 的数据表明，在 BrowseComp 评估中 **Token 使用量解释了 95% 性能方差中的 80%**。单个代理的上下文窗口是硬天花板，而多智能体通过分配独立上下文窗口来突破这一限制。

**agent-swarm 现状验证**：`SubRollout` 的独立 `messages` 列表正是这一设计的体现——每个子代理有自己的完整对话上下文，互不干扰。这个设计方向是正确的。

#### 论据二：并行探索降低路径依赖

研究任务是动态的、路径依赖的。单代理的串行搜索意味着早期的错误方向会浪费后续所有步骤。多智能体的并行探索允许不同代理走不同路径，然后由编排器选择最佳结果。

**agent-swarm 现状验证**：`TaskTool.description` 中已经写了 `"You can launch multiple agents concurrently whenever possible"`。但目前 `execute_tool_calls` 是**串行执行**的（逐个 `for tc in tool_calls`），这是一个关键瓶颈。

#### 论据三：性能提升显著且可量化

Anthropic 的数据：多智能体系统（Opus 4 主 + Sonnet 4 子）比单一 Opus 4 性能**高出 90.2%**。

**agent-swarm 现状验证**：`result/` 目录下保存了 `maink2.5_subk2.5.jsonl` 和 `maink2.5_subqwen.jsonl` 的对比实验结果，说明项目已经在关注异构模型的性能差异。这个方向值得深入。

#### 论据四：成本与价值的平衡

多智能体架构的 Token 消耗约为普通聊天的 **15 倍**。因此它适合**任务价值足够高**的场景（复杂研究、代码生成、数据分析），而非简单问答。

**agent-swarm 现状验证**：当前框架缺乏按复杂度调整投入的机制。无论查询简单还是复杂，都走相同的多智能体流程，存在资源浪费风险。

### 2.2 必要性结论

> agent-swarm 选择多智能体架构的方向是正确的，尤其是其编排器-工作器模式和异构模型支持。但在**并行执行、动态缩放、上下文管理、错误恢复**等关键环节与 Anthropic 生产系统存在显著差距。下文逐一分析。

## 四、代码层面的具体优化建议

### 优化 1：并行执行子代理（严重优先级）

**问题**：`Agent.execute_tool_calls()` 用 `for` 循环串行执行所有工具调用。当 LLM 同时请求启动 3 个子代理时，它们是**依次等待**而非并行运行的。

**当前代码** (`agent/agent.py:151-185`)：
```python
async def execute_tool_calls(self, tool_calls):
    results = []
    for tc in tool_calls:  # ← 串行！
        result = await self.execute_tool(tool_name, arguments)
        results.append(...)
    return results
```

**优化方案**：使用 `asyncio.gather` 并行执行：

```python
import asyncio

async def execute_tool_calls(self, tool_calls):
    async def _execute_single(tc):
        tool_name = tc["function"]["name"]
        args_str = tc["function"]["arguments"]
        try:
            arguments = json.loads(args_str) if isinstance(args_str, str) else args_str
        except json.JSONDecodeError:
            arguments = {}
        result = await self.execute_tool(tool_name, arguments)
        return {
            "tool_call_id": tc.get("id", tool_name),
            "role": "tool",
            "content": result.to_str(),
        }

    # 并行执行所有工具调用
    results = await asyncio.gather(
        *[_execute_single(tc) for tc in tool_calls],
        return_exceptions=True
    )

    # 处理异常
    final_results = []
    for i, r in enumerate(results):
        if isinstance(r, Exception):
            final_results.append({
                "tool_call_id": tool_calls[i].get("id", "error"),
                "role": "tool",
                "content": f"Error: {str(r)}",
            })
        else:
            final_results.append(r)

    return final_results
```

**预期收益**：Anthropic 报告并行工具调用将研究时间**缩短了多达 90%**。

### 优化 2：外部 Memory 机制（高优先级）

**问题**：长对话中 `messages` 列表无限增长，最终超出上下文窗口。Anthropic 的做法是将已完成的工作阶段总结后存储到外部 Memory。

**当前代码无此机制。**

**优化方案**：在 `BaseRollout` 中增加 Memory 管理：

```python
# 新文件: utils/memory.py
import json
from dataclasses import dataclass, field
from typing import List, Dict, Any

@dataclass
class MemoryEntry:
    """外部记忆条目"""
    phase: str           # 阶段名称
    summary: str         # 已完成工作的摘要
    key_findings: list   # 关键发现
    step_range: tuple    # 对应的步骤范围

class AgentMemory:
    """代理外部记忆管理器"""

    def __init__(self, max_context_messages: int = 40):
        self.entries: List[MemoryEntry] = []
        self.max_context_messages = max_context_messages

    def should_compress(self, messages: List[Dict]) -> bool:
        """判断是否需要压缩上下文"""
        return len(messages) > self.max_context_messages

    def compress(self, messages: List[Dict], agent) -> List[Dict]:
        """压缩历史消息，保留 system + 最近消息 + Memory 摘要"""
        if not self.should_compress(messages):
            return messages

        system_msg = messages[0]  # 保留 system prompt
        recent = messages[-20:]   # 保留最近 20 条

        # 中间部分总结为 Memory
        middle = messages[1:-20]
        summary = self._summarize(middle)
        self.entries.append(MemoryEntry(
            phase=f"phase_{len(self.entries) + 1}",
            summary=summary,
            key_findings=[],
            step_range=(0, len(middle)),
        ))

        # 构建压缩后的消息列表
        memory_msg = {
            "role": "user",
            "content": f"[Memory] 以下是之前研究阶段的摘要：\n{self._format_entries()}"
        }
        return [system_msg, memory_msg] + recent

    def _summarize(self, messages: List[Dict]) -> str:
        """将消息列表总结为摘要（简单实现，可接 LLM）"""
        contents = [m.get("content", "")[:200] for m in messages if m.get("content")]
        return f"已完成 {len(messages)} 轮对话，涉及：" + "; ".join(contents[:5])

    def _format_entries(self) -> str:
        return "\n".join(f"- {e.phase}: {e.summary}" for e in self.entries)
```

在 `MainRollout.run()` 的循环中加入压缩检查：

```python
# 在主循环开头增加
if hasattr(self, 'memory') and self.memory.should_compress(self.messages):
    self.messages = self.memory.compress(self.messages, agent)
```

### 优化 4：错误恢复与检查点（高优先级）

**问题**：当前 `MainRollout` 中任何异常都会直接终止整个流程。Anthropic 强调代理需要从错误中恢复。

**当前代码** (`rollout/main_rollout.py:113-121`)：
```python
except Exception as e:
    result = RolloutResult(
        status=RolloutStatus.ERROR, ...
    )
    break  # ← 直接终止
```

**优化方案**：增加容错和检查点机制：

```python
# 在 BaseRollout 中增加
@dataclass
class RolloutConfig:
    max_steps: int = 50
    max_consecutive_errors: int = 3   # 新增：连续错误容忍次数
    checkpoint_interval: int = 10      # 新增：检查点间隔
    # ...

class BaseRollout(ABC):
    def __init__(self, config=None):
        # ...
        self.consecutive_errors = 0
        self.checkpoints: List[Dict] = []

    def _save_checkpoint(self):
        """保存当前状态为检查点"""
        self.checkpoints.append({
            "step": self.current_step,
            "messages_count": len(self.messages),
            "messages_snapshot": self.messages.copy(),
        })

    def _restore_last_checkpoint(self) -> bool:
        """回滚到最近的检查点"""
        if not self.checkpoints:
            return False
        checkpoint = self.checkpoints[-1]
        self.messages = checkpoint["messages_snapshot"]
        self.current_step = checkpoint["step"]
        return True
```

在 `MainRollout.run()` 中使用：

```python
except Exception as e:
    self.consecutive_errors += 1
    logger.warning(f"Error in step {self.current_step} ({self.consecutive_errors}/{self.config.max_consecutive_errors}): {e}")

    if self.consecutive_errors >= self.config.max_consecutive_errors:
        # 连续错误过多，终止
        result = RolloutResult(status=RolloutStatus.ERROR, ...)
        break

    # 添加错误提示，让代理自行调整
    self.messages.append({
        "role": "user",
        "content": f"[System] 上一步执行出错: {str(e)}. 请尝试其他方法继续完成任务。"
    })
    self.current_step += 1
    continue  # ← 继续而非终止
```

class VerifyTool(BaseTool):

### 优化 8：完善日志与可观测性（低优先级但重要）

**问题**：当前日志零散，缺乏结构化追踪。Anthropic 强调监控代理的**决策模式和交互结构**。

**优化方案**：增加结构化事件追踪：

```python
# 新文件: utils/tracer.py
import time
import json
from dataclasses import dataclass, field
from typing import List, Dict, Any

@dataclass
class TraceEvent:
    timestamp: float
    event_type: str     # "llm_call", "tool_exec", "subagent_spawn", "error", "decision"
    agent_id: str
    step: int
    data: Dict[str, Any] = field(default_factory=dict)

class RolloutTracer:
    """结构化执行追踪器"""

    def __init__(self):
        self.events: List[TraceEvent] = []
        self.start_time = time.time()

    def log(self, event_type: str, agent_id: str, step: int, **data):
        self.events.append(TraceEvent(
            timestamp=time.time() - self.start_time,
            event_type=event_type,
            agent_id=agent_id,
            step=step,
            data=data,
        ))

    def summary(self) -> Dict[str, Any]:
        """生成执行摘要"""
        return {
            "total_events": len(self.events),
            "total_time": time.time() - self.start_time,
            "llm_calls": sum(1 for e in self.events if e.event_type == "llm_call"),
            "tool_executions": sum(1 for e in self.events if e.event_type == "tool_exec"),
            "subagents_spawned": sum(1 for e in self.events if e.event_type == "subagent_spawn"),
            "errors": sum(1 for e in self.events if e.event_type == "error"),
        }

    def to_jsonl(self, path: str):
        with open(path, "w") as f:
            for e in self.events:
                f.write(json.dumps({
                    "ts": round(e.timestamp, 3),
                    "type": e.event_type,
                    "agent": e.agent_id,
                    "step": e.step,
                    **e.data
                }, ensure_ascii=False) + "\n")
```

## 附：关键结论

1. **agent-swarm 的架构方向是正确的**——编排器-工作器模式、异构模型支持、防递归设计都与 Anthropic 的生产实践高度吻合。

2. **最大的差距在"并行执行"**——这是一行 `asyncio.gather` 就能解决的问题，但影响巨大。Anthropic 报告并行化将研究时间缩短了 90%。

3. **从实验走向可用的关键是"容错"**——当前遇错即停的设计不适合长时间运行的研究任务。检查点 + 连续错误阈值 + 错误提示是最小可行的容错方案。

4. **动态缩放是成本控制的核心**——多智能体的 Token 消耗是聊天的 15 倍。没有缩放机制意味着每个查询都消耗最大资源，这在实际使用中是不可接受的。

5. **外部 Memory 是规模化的前提**——随着任务复杂度增加，消息列表会快速膨胀。没有 Memory 压缩机制，框架无法处理真正复杂的研究任务。
