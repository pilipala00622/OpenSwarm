# Claude Code Agent Teams 机制总结

> 原文：[Orchestrate teams of Claude Code sessions](https://code.claude.com/docs/en/agent-teams)
> 主题：Claude Code 中 `agent teams` 的工作机制、适用场景与使用边界

---

## 一、这是什么

`Agent teams` 是 Claude Code 提供的一种多智能体协作机制：让多个独立的 Claude Code 会话组成一个团队，共同处理同一项复杂任务。

它的核心不是“单个助手调用几个临时子任务”，而是建立一个**真正可协作的团队结构**：

- 一个 `team lead` 作为主控会话
- 多个 `teammates` 作为独立工作会话
- 一份共享的 `task list` 协调任务领取和完成状态
- 一个 `mailbox` 负责成员之间的消息传递

因此，`agent teams` 更像“一个有分工、有通信、有共享任务板的项目组”，而不只是普通的并行调用。

---

## 二、它解决的核心问题

这个机制主要解决的是：**当任务可以拆成多个相对独立的方向，并且这些方向之间还需要交流时，单一会话效率不够高的问题。**

典型收益有三类：

### 1. 并行探索

多个 teammate 可以同时研究不同方向，而不是主会话一条线顺序推进。

例如：

- 前端、后端、测试并行推进
- 多个 reviewer 分别看安全、性能、测试覆盖
- 多个 debugger 同时验证不同根因假设

### 2. 降低单线程思维偏差

单个 agent 常常会先抓住一个 plausible theory，然后沿着那条线越走越深。  
Agent teams 允许多个成员从不同视角同时推进，并互相质疑，从而减少“过早收敛”。

### 3. 支持协作而非仅汇报

这是它和 subagent 最大的区别。  
Subagent 通常只把结果回传给主 agent；而 teammate 之间可以**直接通信**，共享发现、互相纠偏、共同推进后续任务。

---

## 三、它和 Subagents 的本质区别

官方文档里最重要的一张表，实质上可以概括成下面这句话：

**Subagent 适合“把活分出去做完再回来汇报”；Agent teams 适合“几个人边做边协商”。**

### Subagents

- 各自有独立上下文窗口
- 结果主要回传给主 agent
- 协调中心始终是主 agent
- 更适合短、窄、明确的任务
- Token 成本更低

### Agent teams

- 每个 teammate 也是独立上下文窗口
- 成员之间可以直接发消息
- 通过共享任务列表自协调
- 更适合需要讨论、协同、挑战彼此结论的复杂任务
- Token 成本显著更高

### 直观理解

如果你的任务只是：

- “去查一下这个函数在哪定义”
- “帮我跑一下测试并汇报结果”
- “验证一下某个假设是否成立”

通常 subagent 就够了。

如果你的任务是：

- “多人分别看不同层，然后汇总方案”
- “多人并行调查 bug，并互相反驳对方假设”
- “多人分别负责 feature 的不同模块，还需要持续协调”

那么 agent teams 才真正发挥价值。

## 四、内部机制怎么运作

从机制角度看，`agent teams` 由四个核心部件组成。

### 1. Team lead

主会话，也就是用户当前主要在对话的 Claude Code 实例。

它负责：

- 创建团队
- 生成或拆分任务
- 生成 teammate
- 分配任务或允许成员自领任务
- 汇总结果
- 最终清理团队资源

它不是简单的“总入口”，而是整个团队的协调者。

### 2. Teammates

每个 teammate 都是一个**完整、独立的 Claude Code 会话**。

它们有自己的：

- 上下文窗口
- 工作状态
- 工具调用过程
- 终端交互界面

重要的是：teammate **不会继承 lead 的对话历史**。  
它们会像一个新开的 Claude Code 会话一样加载项目环境，包括：

- `CLAUDE.md`
- MCP servers
- skills
- lead 在 spawn 时给它的 prompt

这意味着：如果某些关键背景只存在于 lead 的对话上下文，而没有写进 spawn prompt，teammate 是不知道的。

### 3. Shared task list

团队共享一份任务列表，任务有三种主要状态：

- `pending`
- `in progress`
- `completed`

任务之间还可以有依赖关系：

- 如果某个 task 依赖另一个 task
- 被依赖项未完成时，它不能被领取
- 一旦依赖完成，系统会自动解除阻塞

文档特别提到，task claiming 使用了**文件锁**来避免多个 teammate 同时抢到同一任务的竞态条件。

这说明它本质上不是“靠模型自己口头约定不要撞车”，而是有一层本地共享状态在做协调。

### 4. Mailbox

团队成员之间通过 mailbox 通信。

支持两种主要消息方式：

- `message`：发给某个指定 teammate
- `broadcast`：发给所有 teammates

而且消息是自动投递的，lead 不需要轮询。

这很关键，因为这让团队具备了“持续协作”能力，而不是一轮式委派。

## 五、一次完整任务的典型流程

可以把它理解为下面这条执行链：

```text
用户提出复杂任务
  -> lead 判断适合并行协作
  -> 用户批准创建 team
  -> lead 生成 teammates
  -> lead 创建/拆分 task list
  -> teammates 领取任务并并行工作
  -> teammates 通过 mailbox 互相沟通或向 lead 汇报
  -> 依赖任务自动解锁
  -> lead 汇总结果并决定下一步
  -> 任务结束后 lead 负责 cleanup
```

这里的关键不是“并行”本身，而是：

- 有共享任务板
- 有自动消息通道
- 有依赖管理
- 有 lead 统一收敛结果

这四者组合起来，才构成 agent teams 的机制闭环。

---

## 六、为什么说它比单会话更重

因为每个 teammate 都是一个独立 Claude 实例，所以它的成本和复杂度都比 subagent 或单会话高得多。

### 1. Token 成本高

每个 teammate 都有自己的上下文窗口和推理过程，因此成本会随着活跃成员数增长。

官方明确强调：

- 对于 research、review、新 feature 这类任务，额外成本通常值得
- 对于例行任务，单一会话更划算

### 2. 协调成本高

成员越多，越容易出现：

- 任务划分不合理
- 信息重复
- 成员等待依赖
- 文件冲突
- lead 过早判断“任务完成”

因此它不是“成员越多越好”，而是只有在**并行收益大于协调开销**时才值得用。

---

## 七、最适合的任务类型

官方推荐的高价值场景主要有四类。

### 1. 跨层改动

比如一个需求同时涉及：

- frontend
- backend
- tests

这时可以让不同 teammate 各自负责一个层面，再由 lead 统一整合。

### 2. 多假设并行调试

当 bug 根因不明确时，不同 teammate 可以各自验证不同理论，并彼此挑战结论。

这种场景非常符合 agent teams 的优势，因为它天然抵抗单一路径依赖。

### 3. 新模块/新功能拆分实现

如果不同模块之间耦合较低，可以让多个 teammate 并行推进各自部分。

### 4. 研究与评审

例如：

- 多 reviewer 从不同标准审 PR
- 多 teammate 分别研究不同技术路线
- 多人并行收集信息再合成结论

这类场景往往边界清楚、文件冲突少、并行收益高。

---

## 八、不适合的任务类型

文档同样强调了很多反例，归纳起来主要是下面几类。

### 1. 强顺序依赖任务

如果 B 一定要等 A 做完，C 又一定要等 B，那么并行收益很低。  
这种情况更适合单会话逐步推进。

### 2. 多人改同一文件

这是最现实的问题之一。  
两个 teammate 改同一个文件，很容易出现覆盖和冲突。

所以实际使用时最好按文件边界切分任务，而不是只按功能描述切分。

### 3. 太小的任务

如果任务只是一个很小的检查、一个简单修改、一个快速验证，那么启动团队的开销超过收益。

### 4. 太多依赖的任务

如果每个步骤都强依赖另一个步骤，团队会大量时间花在等待和协调上。

---

## 九、用户如何控制这个团队

虽然这是一个“团队”，但用户控制点始终在 lead。

### 1. 创建团队

有两种触发方式：

- Claude 主动建议创建 team，但必须经过用户确认
- 用户直接要求创建一个 agent team

文档明确说明：**没有用户批准，不会自动建队。**

### 2. 指定成员数量和模型

用户可以直接要求：

- 建几个 teammate
- 每个 teammate 用什么模型

也可以不指定，让 Claude 自己决定。

### 3. 要求先做计划再实现

对于高风险任务，可以要求 teammate 先进入只读 plan mode，提交方案，由 lead 审批通过后再开始改代码。

这是一个很有价值的机制，因为它相当于把“先设计后落地”内置进团队流程里。

### 4. 直接和某个 teammate 对话

用户不必永远通过 lead 间接沟通。

- 在 split-pane 模式下，可以直接点进某个 pane
- 在 in-process 模式下，可以切换到某个 teammate 并直接发消息

这意味着团队中的每个成员都不是黑盒 worker，而是可直接干预的独立会话。

## 十、显示与交互模式

Agent teams 支持两种显示模式。

### 1. Split panes

每个 teammate 单独占一个 pane，可以同时看到多个成员的输出。  
适合需要频繁观察全员状态的情况。

依赖：

- `tmux`
- 或 iTerm2 + `it2` CLI

### 2. In-process

所有 teammate 运行在当前主终端里，通过快捷键轮换查看。  
优点是几乎无额外环境要求，任何终端都能工作。

默认模式是 `auto`：

- 如果本来就在 tmux 中，优先 split panes
- 否则使用 in-process

## 十一、为什么任务列表是整个系统的关键

如果只看表面，很多人会以为 agent teams 的重点是“能开多个窗口”。  
但真正重要的其实是**共享任务列表**。

因为它承担了三件事：

### 1. 协调分工

谁在做什么，谁还没做，哪些任务卡住，一目了然。

### 2. 管理依赖

任务完成后自动解锁后续任务，减少人工同步成本。

### 3. 支持自领取任务

teammate 做完手头工作后，可以自己去拿下一个未阻塞、未分配的任务，不必每一步都等 lead 明确派发。

这使它从“主控逐项下命令”提升成“带有一定自治能力的协作系统”。

---

## 十二、清理与关闭机制

这部分很容易被忽视，但实际上是系统可靠性的重要一环。

### 1. 关闭某个 teammate

lead 可以向某个 teammate 发送 shutdown 请求。  
teammate 可以接受，也可以拒绝并说明原因。

这说明 shutdown 不是简单粗暴地杀掉进程，而是尽量优雅退出。

### 2. Cleanup 只能由 lead 做

团队结束后应让 lead 执行 cleanup。

官方特别强调：

- 不要让 teammate 自己 cleanup
- 因为 teammate 的 team context 解析可能不完整
- 可能导致共享资源处于不一致状态

### 3. 如果还有活跃 teammate，cleanup 会失败

因此正确顺序通常是：

1. 先关闭所有 teammate
2. 再由 lead 清理团队资源

---

## 十三、最佳实践

结合文档内容，可以总结出几条最关键的实践建议。

### 1. Spawn prompt 一定要补足上下文

因为 teammate 不继承 lead 的完整对话历史，所以必须把任务专属背景写进 prompt。

不要假设 teammate “自然知道你之前聊过什么”。

### 2. 从 3 到 5 个 teammate 起步

官方建议大多数工作流从 `3-5` 个成员开始。  
太少并行度不足，太多则协调成本迅速上升。

### 3. 每个 teammate 最好有 5-6 个任务可做

这样可以减少空转和频繁上下文切换，也方便 lead 在有人卡住时重新分配。

### 4. 先从 research/review 任务入手

这是最容易体现收益、同时最不容易引发文件冲突的场景。

### 5. 切分时优先按文件边界或职责边界拆

不要让两个 teammate 同时碰同一文件。

### 6. 用户要持续观察和纠偏

Agent teams 不适合“开了就不管”。  
如果长时间无人干预，浪费算力和走偏的概率都会增加。

---

## 十四、已知限制

当前这个功能还是 experimental，文档列出了不少限制，值得单独记住。

### 1. 不能嵌套团队

teammate 不能再创建自己的 team。只有 lead 能管理团队。

### 2. 一个会话同一时间只能管理一个 team

必须 cleanup 完当前团队，才能开始下一个。

### 3. Lead 身份不能转移

创建团队的那个 session 会一直是 lead，不能把某个 teammate 提升为 lead。

### 4. 权限继承是从 lead 一次性拷贝的

成员启动时继承 lead 的权限模式；启动后可以改个体模式，但**不能在 spawn 时对不同 teammate 设不同权限模式**。

### 5. 关闭可能较慢

teammate 会等当前请求或工具调用做完再退出，所以 shutdown 不一定即时。

### 6. 任务状态有时会滞后

可能出现任务其实做完了，但没有及时标记 completed，导致后续依赖任务一直被阻塞。

### 7. In-process teammate 无法随 `/resume` 一起恢复

恢复会话后，lead 可能还试图给已不存在的 teammate 发消息，这时需要重新生成 teammate。

### 8. Split-pane 支持环境有限

并不是所有终端都支持，尤其依赖 tmux 或 iTerm2 相关能力。

---

## 十五、如何用一句话理解这套机制

如果只用一句话概括：

**Agent teams = 一个主控 Claude 会话 + 多个可直接互相通信的独立 Claude 会话 + 一套共享任务板与依赖管理机制。**

它的价值不只是“并行”，而是把并行、协作、任务状态和结果收敛放进了同一个系统里。

---

## 十六、我的理解：它真正适合什么

从工程视角看，这个机制最适合的是“**可并行拆分、但又需要协作收敛**”的问题。

如果任务只是简单拆分且彼此不用交流，那么 subagents 更轻。  
如果任务根本拆不开，那么单会话更直接。  
只有当任务满足下面这个条件时，agent teams 才最有价值：

> 各成员能独立推进一段时间，但又需要共享中间发现、协调依赖，并最终由 lead 收敛结果。

这类任务包括：

- 复杂代码审查
- 大范围重构方案研究
- 多假设并行排障
- 跨模块 feature 交付
- 多路线技术选型

换句话说，它不是“Claude 多开”，而是“Claude 的轻量项目组编排器”。

---

## 十七、简明结论

`Agent teams` 的机制重点可以归纳为四点：

1. **结构上**：由 lead、teammates、共享 task list、mailbox 组成。
2. **协作上**：成员既能并行工作，也能彼此通信，而不只是向主 agent 单向汇报。
3. **控制上**：用户仍然通过 lead 掌控建队、审批、分工、关闭和 cleanup。
4. **边界上**：它适合高价值、可并行、需要协同的复杂任务，不适合简单、顺序性强或高冲突的工作。

所以，理解它的关键不是把它看成“更多 agent”，而是把它看成一个**内建任务编排和协作协议的多会话系统**。
