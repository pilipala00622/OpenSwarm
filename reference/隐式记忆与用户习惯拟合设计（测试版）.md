# 隐式记忆与用户习惯拟合设计（测试版）

> 目标：为 `agent-swarm` 这类 Agent 框架设计一套“无意识 memory 养成”机制，让系统在长期协作中逐步拟合用户习惯，并据此更自然地展开工作。

---

## 一、这份设计想解决什么问题

当前很多 Agent 系统的 memory，大多停留在两类：

- **会话记忆**：保留当前对话历史
- **摘要记忆**：把历史任务压缩成摘要，存起来供后续参考

这两类都很有价值，但还不够接近理想状态。

更理想的体验是：

- 用户不需要频繁显式配置偏好
- 系统能从长期互动中逐步理解“这个人通常怎么工作”
- 这种理解不是一次性写死，而是慢慢拟合、逐步增强、可被修正

也就是说，系统记住的不只是“发生过什么”，还包括：

- 用户偏好什么工作方式
- 用户默认更在意什么
- 用户在不同场景下通常如何做取舍
- 哪些偏好是全局的，哪些只在当前项目成立

这就是这里说的**隐式记忆**。

---

## 二、什么叫“隐式记忆”

隐式记忆不是用户明确说“请记住这个”，而是系统通过长期观察，逐步归纳出来的稳定偏好。

例如：

- 用户每次都先要分析，再允许动手修改
- 用户做 review 时，总是优先关注风险而不是总结
- 用户写代码时，更偏好小步迭代而不是大改
- 用户解释问题时，更喜欢先给结构，再展开细节

这些模式未必被用户显式表达过，但会在持续互动中反复出现。

如果系统能稳定识别这些重复模式，就可以在后续任务中把它们作为默认工作方式的一部分。

---

## 三、核心判断：理想的 memory 不是知识库存，而是工作风格拟合

传统 memory 更像“记事本”，记录的是：

- 某次任务做了什么
- 某个项目有哪些事实
- 某些上下文是否可以复用

而这里想要的系统更像一个**用户工作风格模型**。

它回答的不是：

- “历史上发生了什么？”

而是：

- “这个用户通常希望怎么推进工作？”
- “遇到代码修改任务时，他更偏好什么顺序？”
- “在当前项目里，他默认更在意规范、速度还是稳健性？”

所以这类 memory 的核心，不是存更多文本，而是形成一组**可激活的偏好分布**。

---

## 四、建议的记忆分层

要做到“拟合用户习惯 + 无意识养成”，memory 不能只有一层，建议至少拆成下面五层。

### 1. Rule 层

这是最高优先级的规则层，特点是明确、稳定、可覆盖。

例如：

- 不要在未确认前执行 destructive 操作
- review 默认先找问题，再给总结
- 不要自动提交未请求的 git commit

这一层不是“偏好”，而是约束。

### 2. Explicit Memory 层

用户明确说“请记住”的内容，适合记录长期稳定偏好。

例如：

- 我更喜欢先看结论再看细节
- 我默认希望你用中文回答
- 我更喜欢小步提交

这一层是**用户显式授权的长期记忆**。

### 3. Implicit Memory 层

系统从长期行为中自动归纳出来的弱偏好。

例如：

- 该用户在 80% 的代码任务中都先要求分析影响范围
- 该用户在 review 类任务中更常关注风险和缺失测试
- 该用户经常要求把方案写成可落地设计稿

这一层是本设计的核心。

### 4. Project Memory 层

只在当前仓库、当前项目或当前团队内生效的记忆。

例如：

- 这个项目偏好最小侵入式修改
- 这个仓库文档以中文为主
- 这个项目强调 demo、reference 和 README 同步更新

这类记忆不能轻易外溢到其他项目。

### 5. Session Intent 层

这是当前任务的短期状态，不应该沉淀成长期习惯。

例如：

- 这一轮先只写方案，不要改代码
- 这次先快速探索，不要求完整验证
- 当前只需要产出一份测试版文档

这一层随任务结束而衰减，不应长期保留。

## 五、隐式记忆如何“养成”

这里的关键不是“自动记录一切”，而是**只对重复、稳定、跨场景出现的模式逐步增强权重**。

### 1. 先采集行为信号

每轮交互后，不是只总结任务本身，还要提取一些行为特征，例如：

- 用户更常要求“先分析后执行”还是“直接动手”
- 用户偏好长解释还是短解释
- 用户更常要求风险说明、测试说明还是实现说明
- 用户偏好一步到位还是分阶段推进
- 用户是否常常强调边界条件、回滚、可观测性

### 2. 将行为信号转成候选偏好

不能直接把一次行为写成 memory，而是先记成候选项。

例如：

- `pref.analysis_before_action += 1`
- `pref.concise_explanation += 1`
- `pref.risk_first_review += 1`

也就是说，先积累证据，不立即下结论。

### 3. 通过阈值与时间窗口升格

只有当某个偏好满足以下条件，才可以升格为隐式记忆：

- 多次重复出现
- 间隔较长时间仍然成立
- 在不同任务类型中都出现
- 没有被用户明确反驳

这一步的目标是避免把一次临时行为误学成长期偏好。

### 4. 用“权重”而不是“真值”表示

隐式记忆不应该写成硬编码事实，例如：

- `user_prefers_analysis_first = true`

更合理的形式应该是：

- `analysis_before_action: confidence=0.76`
- `risk_first_review: confidence=0.83`
- `concise_delivery: confidence=0.58`

这样系统才能保留弹性，根据任务场景决定是否激活。

## 六、激活逻辑：不是每轮都注入全部记忆

成熟的 memory 系统不应该在每次任务里都塞入全部历史偏好。

更合理的方式是：

### 1. 先判断当前任务类型

例如判断这是：

- 代码实现任务
- 代码 review 任务
- 文档总结任务
- 架构讨论任务
- 调试/排障任务

### 2. 再激活相关偏好

例如：

- review 任务优先激活“风险优先”“测试覆盖关注”
- 文档任务优先激活“先结构后细节”“更偏中文表达”
- 改代码任务优先激活“小步修改”“先分析影响范围”

### 3. 控制注入强度

不是所有记忆都要写进主 prompt，可以拆成不同注入方式：

- 高优先级规则 -> `System Prompt`
- 当前轮重要提醒 -> `System Reminder`
- 相关偏好摘要 -> 动态轻量注入
- 历史证据详情 -> 仅在需要时检索

这和上下文优化的思路是完全一致的：

**记忆也应该按需暴露，而不是全量常驻。**

## 七、需要特别警惕的风险

### 1. 把临时偏好误学成长期习惯

这是最大的风险。

例如用户只是在当前任务里说了一次“先快点做”，系统却把它学成“长期更重速度”。

### 2. 用户风格被历史记忆绑死

如果系统过度依赖旧偏好，就会在用户已经改变工作方式时表现得很僵硬。

所以记忆必须支持：

- 衰减
- 覆盖
- 删除
- 场景化失活

### 3. 项目记忆污染全局记忆

用户在 A 项目里的习惯，不一定能迁移到 B 项目。

因此全局与项目记忆必须严格隔离。

### 4. 自动记忆缺乏可解释性

如果系统已经“学会了某种习惯”，但用户不知道它为何这么做，就会很难建立信任。

所以系统最好能解释：

- 当前激活了哪些偏好
- 这些偏好来自显式记忆还是隐式归纳
- 置信度大概有多高

## 八、一个更理想的工程机制

如果把这套东西做成产品能力，我会倾向于这样设计：

### 1. 弱记忆阶段

先只在后台记录候选偏好，不直接影响行为。

### 2. 试探性激活阶段

当某项偏好积累到一定置信度后，开始轻量影响行为，但不要太强。

例如：

- 优先在解释顺序上体现
- 优先在任务拆解方式上体现
- 优先在输出组织上体现

### 3. 稳定记忆阶段

当某项偏好多次被验证为有效，才把它升格为稳定 memory。

### 4. 用户纠偏阶段

允许用户显式地说：

- 这个偏好不要记
- 这个偏好只在当前项目生效
- 这个偏好最近已经不成立

这一步非常重要，否则“无意识养成”最后会变成“无意识固化”。

## 九、落到 `agent-swarm` 的一个可行演进路径

如果把这个方向落到 `agent-swarm`，我会建议分三步走。

### 第一阶段：从摘要记忆升级到结构化记忆

把现有 memory 从单一摘要，拆成几类：

- `task_memory`
- `project_memory`
- `user_preference_memory`
- `session_intent`

这一步的目标是先把不同性质的信息分开。

### 第二阶段：增加隐式偏好提取

每轮交互结束后，除了做任务总结，还提取轻量偏好信号，例如：

- 是否要求先分析
- 是否偏好简洁输出
- 是否强调边界条件
- 是否经常要求对比方案

先把这些写成候选偏好，不直接强影响系统行为。

### 第三阶段：做场景化激活

在新任务开始时：

- 先识别任务类型
- 再从 memory 中挑选相关偏好
- 生成一段短的 reminder 注入当前轮

这样既能保持个性化，也不会把 prompt 撑爆。

---

## 十、一个测试版定义

如果要给这个方向下一个简短定义，我会这样写：

**隐式记忆不是把历史记录得更多，而是让系统逐步学会用户稳定的工作方式，并在合适的场景中自然激活。**

它的目标不是“替用户保存所有信息”，而是：

- 降低重复表达成本
- 提升协作默契
- 让 agent 更像一个持续磨合过的搭档

---

## 十一、当前测试版结论

你理想中的状态，本质上不是一个“更大的 memory”，而是一套：

- 分层的记忆结构
- 渐进的偏好拟合机制
- 可纠偏的长期学习回路
- 按任务场景激活的动态注入系统

如果这一套做对了，memory 就不再只是“记住内容”，而会逐步变成：

**系统对用户工作风格的长期建模。**
