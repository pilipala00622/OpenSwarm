# Quest 1.0 上下文与长期记忆总结

> 原文 1：[Qoder - The Agentic Coding Platform](https://qoder.com/blog/quest-context-optimize)
> 原文 2：[Qoder - The Agentic Coding Platform](https://qoder.com/blog/long-term-memory)
> 参考文档：[Memory - Qoder](https://docs.qoder.com/user-guide/chat/memory)
> 主题：Quest 1.0 如何同时优化上下文装载与长期记忆，让 Agent 在降低 token 成本的同时，保留稳定偏好和项目经验

---

## 一、这篇文章在讲什么

Quest 1.0 的核心观点不是“让模型看到更多”，而是“让模型只看到当前真正需要的东西”。

文章提到有两项上下文优化已经默认上线，用户**不需要额外打开设置**。这说明它不是把优化当成手动调参能力，而是把它做成了 Agent 系统的一层默认基础设施。

这背后的目标很明确：

- 减少无效上下文带来的 token 浪费
- 提高 prompt cache 命中率
- 让工具选择更接近“按需注入”而不是“全量暴露”
- 在不牺牲调用成功率的前提下，让 agent 更轻、更快

---

## 二、为什么这件事重要

很多 AI Agent 系统在变复杂以后，上下文膨胀通常不是来自用户输入，而是来自系统自己塞进去的内容：

- 很长的 system prompt
- 全量工具定义
- 大段运行时状态
- 低质量索引召回的上下文片段

其中最容易被忽略的一类，就是**从来没有被调用过的工具定义**。

也就是说，很多系统一边说自己在做“工具增强”，一边却在每轮对话里持续支付大量和当前任务无关的 token 成本。

Quest 1.0 的优化，本质上是在解决这个问题：

**不要把“工具很多”误当成“能力很强”，真正重要的是“当前时刻暴露给模型的工具是否精准”。**

---

## 三、如何理解文中的几个关键判断

### 1. 先跑 token 分布分析，再决定优化方向

文中的第一个建议是：先看 context token 到底花在了哪里。

这意味着优化不能靠直觉，而要先做分布分析，例如：

- system prompt 占多少
- 工具 schema 占多少
- 动态 reminder 占多少
- 历史消息占多少
- 检索上下文占多少

如果分析后发现大头是工具定义，那么问题不在模型，而在**工具暴露策略**。

这对自建 Agent 很有启发：

- 先测 token 构成
- 再决定该压缩哪一层
- 不要一上来就换模型或缩短用户输入

### 2. 意图匹配不需要完整工具定义

文章强调：模型判断“要不要用某个工具”，往往不需要先读完完整 schema。

也就是说，在工具选择阶段，真正重要的不是长篇参数细节，而是：

- 工具名是否清楚
- 一行描述是否准确
- 它解决的任务边界是否明确

这意味着工具可以分成两层：

- **索引层 / 路由层**：给模型看简短名称和描述，用来判断需不需要
- **执行层 / 完整定义层**：只有真的要调用时，再把完整 schema 注入

这是非常典型的“先检索，再展开”的思路。

### 3. 索引质量决定上限

如果工具索引写得不准，动态注入机制再高级也没用。

因为系统根本不会报错，它只会出现一种更难发现的问题：

**该调用的工具没有被选中。**

所以工具描述的质量，本质上决定了整个工具路由系统的上限。好的描述应该做到：

- 一句话就能说清用途
- 明确输入输出
- 明确适用场景
- 能和相似工具区分开

如果一句描述写偏了，工具不会“异常退出”，而是会“安静失效”。

### 4. 动态内容应放在 System Reminder，而不是 System Prompt

这条建议非常关键。

原因不只是组织结构更清楚，更重要的是**不要破坏 Prompt Cache**。

如果把频繁变化的内容直接拼进 system prompt，会导致缓存基座不断变化，进而降低 cache 命中率。  
而把动态内容放在独立的 reminder 层，可以：

- 保持主 system prompt 尽量稳定
- 让缓存更容易复用
- 把“稳定规则”和“运行时状态”分层管理

这其实是在强调一个工程原则：

**静态指令和动态注入要分层。**

### 5. 动态注入之后一定要回归测试完整工具调用链路

文章提醒：索引描述写偏了不会报错，只会默默不触发。

因此，做完上下文优化以后，不能只看 token 降了多少，还要验证：

- 该用的工具还能不能被选中
- 关键任务下工具调用率是否下降
- 是否出现误召回、漏召回
- 多工具场景下是否还能选对正确工具

这说明工具优化是一个**双目标问题**：

- 一方面要降低上下文成本
- 另一方面不能损失调用正确率

所以最终评估必须同时看：

- token 成本
- cache 命中
- 工具触发率
- 工具选择准确率
- 任务成功率

## 四、把它翻译成自建 AI Agent 的设计原则

如果你在自己搭 AI Agent，这篇文章最值得吸收的是下面几条：

### 1. 工具不要全量常驻

默认把所有工具 schema 都塞进 prompt，短期简单，长期代价非常高。

更合理的方式是：

- 先维护一个轻量工具索引
- 先做意图匹配
- 再按需注入完整工具定义

### 2. 工具描述本身就是“检索索引”

很多团队把工具描述当作文档，其实它更像“给模型看的检索标签”。

所以写工具描述时要把重点放在：

- 这个工具解决什么问题
- 在什么情况下该用
- 它和其他工具怎么区分

### 3. Prompt 架构要分层

推荐至少分成三层：

- **System Prompt**：稳定、通用、不轻易变化的行为规则
- **System Reminder**：当前任务态、动态策略、运行时上下文
- **Tool Injection**：基于意图匹配按需加入的工具定义

这样做能同时改善：

- prompt cache
- 可维护性
- 诊断能力

### 4. 上线后要持续观测，而不是一次性压缩

需要长期监控的指标包括：

- context token 分布
- tool selection 触发率
- 各工具命中率
- 任务完成率
- 错误调用与漏调用

否则你只会看到“token 下降了”，却不知道是不是以“能力下降”为代价换来的。

## 五、再看长期记忆：它解决的是另一类问题

如果说前一篇文章解决的是“**当前轮应该给模型看什么**”，那长期记忆解决的是“**哪些信息不该每轮都重写，却又应该长期保留**”。

从 Qoder 对 memory 的公开描述来看，它的长期记忆至少覆盖两种来源：

- **Active Memory**：用户显式告诉系统“记住这件事”
- **Automatic Memory**：系统从长期交互中自动学习编码习惯、项目偏好和上下文线索

同时，它又有两层作用域：

- **Global Memory**：跨项目生效，更偏向个人习惯与通用偏好
- **Project Memory**：项目内生效，更偏向仓库约定、技术栈与局部工作方式

这说明它不是把 memory 理解成“另一段更长的 prompt”，而是把它理解成一层**可持续沉淀的偏好与经验层**。

## 六、如何理解记忆模式的演化

记忆模式通常不是一步到位的，而是会经历一个很典型的演化路径。

### 1. 第一阶段：会话内记忆

最早的“记忆”其实只是当前对话历史。

特点是：

- 实现简单
- 对当前任务足够
- 一旦开新会话就丢失
- 容易随着上下文变长而失控

这一阶段本质上还不是真正的 memory，只是 conversation state。

### 2. 第二阶段：摘要式外部记忆

再往前一步，系统开始把长对话压缩成摘要，存到外部文件或数据库里。

特点是：

- 比纯会话历史更持久
- 成本更低
- 适合保存任务进度、阶段结论、handoff 信息
- 但仍然偏“事件记录”，不够像稳定偏好

这一阶段解决的是“不要丢上下文”，但还没完全解决“如何保留长期风格和习惯”。

### 3. 第三阶段：显式偏好记忆

系统开始支持用户主动声明：

- 我的代码风格是什么
- 我偏好的测试方式是什么
- 这个项目有哪些固定约定

这就是 `Active Memory` 的价值。  
它让 memory 从“系统自己总结了什么”，进化成“用户可以控制系统长期记住什么”。

### 4. 第四阶段：自动习得记忆

再进一步，系统不只是被动接收，还会从长期交互中自动归纳：

- 你偏好什么风格
- 这个项目默认怎么命名
- 你常用哪些工具和工作流
- 哪些约定是跨会话稳定存在的

这就是 `Automatic Memory` 的方向。  
它的价值在于减少重复表达，让 agent 越用越贴近真实工作方式。

### 5. 第五阶段：分层记忆系统

真正成熟的记忆系统，通常不会只有“一份 memory”，而会拆成多层：

- **规则层**：最高优先级，明确约束 agent 行为
- **全局记忆层**：用户长期稳定偏好
- **项目记忆层**：仓库内约定、历史决策、团队习惯
- **任务态上下文层**：当前任务的动态状态
- **即时检索层**：本轮按需召回的代码、文档、工具

这时候 memory 不再是“存起来就行”，而是变成一个**分层决策系统**。

## 七、为什么长期记忆不能替代上下文优化

这两者看起来都在处理“给模型什么信息”，但其实分工完全不同：

- **上下文优化**：解决“当前轮到底注入哪些内容”
- **长期记忆**：解决“哪些稳定信息应该跨轮保留下来”

如果没有上下文优化，memory 再多也可能把 prompt 塞爆。  
如果没有长期记忆，系统就会不停让用户重复同样的偏好和约定。

所以更合理的理解是：

- `context optimize` 负责**按需暴露**
- `long-term memory` 负责**持续沉淀**

两者结合起来，系统才会既轻量，又连续。

## 八、把长期记忆翻译成自建 Agent 的设计原则

如果你在自己搭 AI Agent，长期记忆最值得借鉴的是下面几点：

### 1. 记忆要区分“规则、偏好、任务态”

不要把这些东西全塞进一个 memory blob。

更合理的划分是：

- **Rules**：强约束，优先级最高
- **Preference Memory**：风格、习惯、偏好
- **Project Memory**：项目内长期事实和约定
- **Task State**：当前任务的临时状态

这也是为什么 Qoder 文档里会强调：当 Rules 和 Memory 冲突时，**Rules 优先**。

### 2. 记忆应该可管理、可删除、可显式写入

如果 memory 只能自动积累，最后一定会污染。

一个健康的记忆系统至少要支持：

- 用户主动写入
- 用户查看当前记忆
- 用户删除错误或过时记忆
- 系统区分全局与项目作用域

### 3. 自动记忆要谨慎，不要把偶发行为学成长期偏好

自动记忆很强，但也最容易出问题。

因为系统很可能把：

- 某次临时妥协
- 某个实验性实现
- 某段仓促指令

错误地沉淀成长期模式。

所以自动记忆通常需要：

- 更高的写入门槛
- 更好的可解释性
- 更明确的编辑和撤销能力

### 4. 记忆不是越多越好，而是越稳定越值钱

真正适合进入长期记忆的，通常不是一次性的工作细节，而是：

- 长期稳定的编码偏好
- 项目内反复复用的规范
- 高概率在未来继续生效的事实

## 九、对 `agent-swarm` 这类框架的启发

这篇文章和 `agent-swarm` 当前的演进方向其实很契合，尤其是两点：

### 1. `retrieve_context` / 知识引擎层的意义更清楚了

知识引擎不只是为了“给更多上下文”，而是为了让上下文**更有选择性地进入 prompt**。

也就是说，知识引擎的价值不只是召回，而是：

- 先过滤
- 先排序
- 先缩小候选集
- 再把高价值上下文交给 agent

### 2. 工具系统未来可以进一步做“轻索引 + 动态展开”

如果后续继续演进 `agent-swarm`，一个自然方向是：

- 给每个工具维护更短的 intent description
- 先做工具级别的轻量匹配
- 只在命中后再展开完整参数 schema

这会非常适合工具逐渐增多的阶段。

### 3. 现有 `memory` 能力可以继续往“分层记忆”演进

`agent-swarm` 现在已经有外部记忆与摘要压缩能力，这很接近“第二阶段：摘要式外部记忆”。

如果继续演进，一个很自然的方向是把 memory 继续拆成：

- **handoff / task summary**：偏任务态
- **user preference memory**：偏用户长期风格
- **project memory**：偏仓库级约定
- **runtime reminder**：偏本轮动态状态

这样可以避免把所有东西都塞进同一种 memory 文档里。

### 4. 可以把“记忆写入”设计成显式工具，而不是隐藏副作用

例如后续可以考虑：

- `remember_preference`
- `remember_project_fact`
- `list_memory`
- `forget_memory`

这样 memory 不只是框架内部的总结结果，而会变成 agent 可控制、可审计的一等能力。

---

## 十、一句话总结

把两篇文章放在一起看，最值得记住的一点是：

**Agent 的演化方向，不是无限堆上下文，而是把信息分成两类处理: 当前需要的内容按需暴露，长期稳定的内容沉淀成记忆。**

真正有效的系统，不是让模型始终看到全部能力和全部历史，而是让它在正确的时候看到正确的工具、上下文和长期偏好。
