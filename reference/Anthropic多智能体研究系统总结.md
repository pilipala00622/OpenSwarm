## 一、概述

Claude 现已具备跨网络、Google Workspace 及各种集成工具进行搜索的研究能力，可以完成复杂的信息调研任务。本文介绍了 Anthropic 将一个多智能体研究系统从原型推向生产环境的完整过程，涵盖系统架构、工具设计和提示工程等核心内容。

**多智能体系统**由多个自主使用工具的 LLM（即"代理"）组成，通过协作完成复杂任务。这种架构在带来强大能力的同时，也引入了协调、评估和可靠性方面的全新挑战。

---

## 二、为什么选择多智能体架构

### 2.1 适应开放性问题

研究任务本质上是动态的、路径依赖的，无法用硬编码的固定路线完成。多智能体系统允许代理根据发现持续调整策略。

### 2.2 并行处理与信息压缩

子代理可以在各自独立的上下文窗口中**并行**探索问题的不同方面，然后将关键信息压缩汇总给主导代理。这种分工有效减少了路径依赖。

### 2.3 性能提升显著

- 使用 **Claude Opus 4** 作为主导代理、**Claude Sonnet 4** 作为子代理的系统，在内部研究评估中比单一 Claude Opus 4 系统性能**高出 90.2%**。
- 在 BrowseComp 评估中，Token 使用量解释了 95% 性能方差中的 **80%**。

### 2.4 成本考量

多智能体架构消耗 Token 的速度极快：
- 代理交互的 Token 消耗约为普通聊天的 **4 倍**
- 多智能体系统约为普通聊天的 **15 倍**

因此，这类系统更适合**任务价值足够高**的场景。

## 三、系统架构：编排器-工作器模式

系统采用**编排器-工作器（Orchestrator-Worker）模式**，由一个主导代理协调整体流程，并将子任务委托给并行运行的专用子代理。

### 完整工作流程

```
用户提交查询
    │
    ▼
[LeadResearcher 主导代理]
    │── 分析查询，制定研究策略
    │── 将计划保存到 Memory（防止上下文截断）
    │
    ▼
[创建多个 Subagents 子代理]
    │── 每个子代理有特定的研究任务
    │── 并行执行：网络搜索 + 交错思维评估
    │
    ▼
[主导代理综合结果]
    │── 判断信息是否充分
    │── 不足则创建更多子代理或调整策略
    │
    ▼
[CitationAgent 引用代理]
    │── 处理文档，识别引用位置
    │── 确保每项声明有据可查
    │
    ▼
带引用的最终结果 → 返回用户
```

## 四、8 条提示工程原则

### 1. 像你的代理一样思考

通过模拟和观察代理的实际行为来理解提示的效果，从而发现潜在的失败模式。

### 2. 教导编排器如何委派

主导代理需要为子代理提供清晰的：
- 任务目标
- 输出格式
- 工具使用指导
- 任务边界

避免不同子代理之间的重复劳动或遗漏。

### 3. 根据查询复杂度调整投入

在提示中嵌入缩放规则，例如：
- 简单任务 → 1 个子代理
- 复杂任务 → 10+ 个子代理

防止在简单问题上浪费资源。

### 4. 工具设计和选择至关重要

为代理提供明确的启发式规则：
- 检查所有可用工具
- 优先使用专用工具
- 避免因工具描述不清导致错误路径

### 5. 让代理自我改进

利用 Claude 4 模型作为"提示工程师"来诊断失败模式并改进工具描述。这一做法使后续代理的**任务完成时间减少了 40%**。

### 6. 先宽后窄

模仿人类专家的研究策略：
- 先用短而广泛的查询建立全局认知
- 再逐步聚焦到具体细节

### 7. 引导思维过程

利用扩展思维（Extended Thinking）模式让代理：
- 规划研究方法
- 评估信息质量
- 识别知识缺口

### 8. 并行工具调用

主导代理并行启动子代理，子代理并行使用工具。这种做法将复杂查询的研究时间**缩短了多达 90%**。

## 五、评估方法

### 5.1 小规模快速评估

在早期开发阶段，即使只有约 20 个查询样本，也能显著发现提示调整带来的巨大影响。**尽早开始评估**是关键。

### 5.2 LLM 作为裁判

使用 LLM 根据以下维度对输出进行评分：
- 事实准确性
- 引用准确性
- 信息完整性
- 来源质量
- 工具使用效率

该方法在数百个输出的大规模评估中表现良好。

### 5.3 人工评估不可替代

人类测试员能发现自动评估遗漏的边缘情况。例如，早期发现代理倾向于选择 SEO 优化的内容农场而非权威来源。

## 六、生产环境的工程挑战

### 6.1 状态性与错误累积

代理在长时间运行中维护状态，错误可能逐步累积。系统需具备：
- **检查点和重试逻辑**：从错误中恢复
- **模型的自适应能力**：优雅处理异常

### 6.2 调试方法革新

- 添加全面的生产追踪
- 监控代理的**决策模式和交互结构**（而非单个对话内容）
- 以此诊断系统级根本原因

### 6.3 彩虹部署（Rainbow Deployment）

采用渐进式部署策略，逐步将流量从旧版本转移到新版本，**避免中断正在运行的代理**。

### 6.4 同步执行的瓶颈

当前的同步执行限制了并行性。未来转向异步执行将带来性能提升，但也增加了结果协调和状态一致性的挑战。

## 七、实用技巧（附录）

### 7.1 最终状态评估

对于修改持久状态的代理，重点评估**最终状态是否正确**，而非纠结于具体的中间步骤。

### 7.2 长对话管理

使用智能压缩和外部内存机制：
- 总结已完成的工作阶段
- 存储到外部 Memory
- 有效管理超长对话

### 7.3 子代理输出到文件系统

允许子代理将结构化输出（代码、报告等）存储在外部系统中，传回轻量级引用给协调器，以减少信息丢失和 Token 开销。

## 九、结论

构建多智能体系统的"最后一公里"往往是最艰难的。尽管如此，该系统在开放式研究任务中已充分证明其价值——用户反馈显示它能发现商业机会、解决复杂技术难题并节省大量时间。

成功的生产部署需要：
- 精心的工程设计
- 全面的测试覆盖
- 细致的提示和工具设计
- 跨团队的紧密协作
